Virtual vs Override vs New Keywords in C#

Introduction

The purpose of writing this article is simple, which is- a simple and fresh demonstration of basic difference between these three mostly used and confusing keywords in C# with some reference example. This article is purely for the beginner in C#.

Outline

Overview

In this article, I’ll explain how virtual, override and new keywords are different from each other by taking some set of examples respectively (some sort of functionality).

At the end, there will be some key points to remember about all these 3 keywords.

Introduction

In this introductory part, I am going to give a brief introduction of all these 3 keywords. So here they are:

Virtual Keyword

Virtual keyword is used for generating a virtual path for its derived classes on implementing method overriding. Virtual keyword is used within a set with override keyword. It is used as:

Hide   Copy Code
// Base Class
    class A
    {
        public virtual void show()
        {
            Console.WriteLine("Hello: Base Class!");
            Console.ReadLine();
        }
    }

Override Keyword

Override keyword is used in the derived class of the base class in order to override the base class method. Override keyword is used with virtual keyword, as:

Hide   Copy Code
// Base Class
    class A
    {
        public virtual void show()
        {
            Console.WriteLine("Hello: Base Class!");
            Console.ReadLine();
        }
    }

// Derived Class

    class B : A
    {
        public override void show()
        {
            Console.WriteLine("Hello: Derived Class!");
            Console.ReadLine();
        }
    }

New Keyword

New keyword is also used in polymorphism concept, but in the case of method overloading So what does overloading means, in simple words we can say procedure of hiding your base class through your derived class.

It is implemented as:

Hide   Copy Code
class A
    {
        public void show()
        {
            Console.WriteLine("Hello: Base Class!");
            Console.ReadLine();
        }
    }

    class B : A
    {
        public new void show()
        {
            Console.WriteLine("Hello: Derived Class!");
            Console.ReadLine();
        }
    }

Let’s see:

Key Points

I am giving some key points about these keywords by taking reference of method overloading and overriding concepts, as these keywords are used in these mechanism.

Virtual & Override

  • Used in polymorphism implementation
  • Includes same method name and same params’
  • Used in method overriding concept
  • It is also called run time polymorphism
  • Causes late binding

New

  • It is also used in polymorphism concept
  • Includes same method name and different params’
  • Used in method overloading concept
  • It is compile time polymorphism
  • Cause early binding

[My vote of 2] Missing some very important points
Assorted Trailmix9-Sep-14 18:39
Assorted Trailmix9-Sep-14 18:39 
First and foremost, as has been mentioned in other comments, "overhiding" which seems to be a portameau of "override" and "hiding" isn't a term. "Hiding" (or more accurately "Name hiding through inheritance") describes what it does.

Secondly, this article is presenting very similar examples to those available all over then internet (and on MSDN), but omitting the single most important part. Excuse that emphasis, but it cannot be stressed enough how important it is to realize you will almost never want to use "new" when you see that warning from the compiler. It's unfortunate so many sources (including the warning itself) neglect to make this clear.

If you look carefully at the definition on the page for "Versioning with the Override and New keywords" you'll see the line "The C# language is designed so that versioning between base and derived classes in different libraries can evolve and maintain backward compatibility".

The new modifer is the embodiment of the "dark side" of versioning. As Eric Lippert explains in this article, it's something that acts as a kind of "band-aid" due to the lack of "return type covariance", which would allow you to return a different type than defined in the base, as long as it could be cast to the type defined in the base:
Hide   Shrink   Copy Code
class BaseClass
{
    IEnumerable<MyClass> MyClassObjects...
}

class DerivedClass : BaseClass
{
    List<MyClass> MyClassObjects... //implements IEnumerable<MyClass>, but we can't do this without a warning
}

When you define a method with the new modifer, you're usually saying that "I have a really good reason why I don't want to change the base member, but I really need to change what it does". But it can make for code that can become very hard to follow.
Consider this, which will compile and run:
Hide   Shrink   Copy Code
void SomeFunc()
{
    BaseClass baseInstance = new DerivedClass();
    baseInstance.MyClassObjects = new HashSet<MyClass>();
}


Now that doesn't make sense does it? After all, you can't do this:
Hide   Shrink   Copy Code
    List<MyClass> myList = new HashSet<MyClass>(); //You can't cast between those two


You should almost always use override. The most common example I see that uses new, which the article actually mentions, is when you have a base class that you already derived, and you need to add generics to an existing member:

Hide   Shrink   Copy Code
class MyClass{}
class MyClass<T> : MyClass{}
class BaseClass : BaseClass//This could already be in use in something that shipped out, changing it could be a breaking change
{
   MyClass GetMyClass(){...}
}

class DerivedClass : BaseClass //Same here
{
    MyClass GetMyClass(){...}
}
class EvenMoreDerivedClass<T> : BaseClass //Maybe this came later
{
   MyClass<T> GetMyClass(){...}//This is the only logical name for this (why use GetMyClass2())
}


The example that the article mentions is an excellent example of this that we all know, IEnumerable (which existed first in pre-generics C#).

Long story short, if you need to look up override vs new, rather than how one works, odds are you don't need new. There are people who have written code in C# for years without needing it. Default to override, and if you need new, it will be very obvious.

You wrote:
"New keyword is also used in polymorphism concept, but in the case of method overhiding."

To me, overhiding sounds like an error when writing overriding. Yet, override does overriding. new does hiding only. There's no "over" there.

If that's not the case, I would like a different explanation, as override and new seem to do: overriding and overhiding and I am not understanding the difference by these words.



Also, in the conclusion, you wrote this:

"Virtual & Override
•Used in polymorphism implementation
•Includes same method name and same params’
•Used in method overriding concept
•It is also called compile time polymorphism
•Causes early binding

New
•It is also used in polymorphism concept
•Includes same method name and different params’
•Used in method overhiding concept
•It is run time polymorphism
•Cause late binding"

Actually, virtual and override are run-time polymorphism, not compile time.
New is compile-time polymorphism, not run-time.
New also doesn't use different parameters. If you have a method with the same name but different parameters, there's no conflict. Putting the new modifier in this situation will cause the warning: The member 'name here' does not hide an inherited member. The new keyword is not required.
